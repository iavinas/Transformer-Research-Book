Chapter 14

Training Language Models at Scale

Training GPT-3's 175 billion parameters requires roughly $3.14 \times 10^{23}$ floating-point operations. A single A100 GPU delivering 312 TFLOPS would need over 36 years. The actual training took 34 days on 1,024 GPUs — and the harder problem was not speed but fit: 175B parameters in mixed-precision Adam consume over 2.8 terabytes of memory, far beyond any single device. Every technique in this chapter attacks one or both of these constraints — fitting the model across devices and keeping those devices busy. By the end, you will understand how BLOOM, LLaMA, and DeepSeek V3 were actually trained, from the ring all-reduce that synchronizes gradients to the 3D parallelism grid that maps a trillion-parameter model onto thousands of GPUs.

Data Parallelism

The simplest way to use multiple GPUs is to give each one a copy of the entire model and a different slice of the training data. Each GPU computes gradients on its own mini-batch, then the gradients are averaged across all GPUs so every copy stays in sync. This is data parallelism, and it remains the foundation of every large-scale training system.

Naive Data Parallelism and All-Reduce

In naive data parallelism, $N$ GPUs each hold a full replica of the model with parameters $\theta$. Each GPU runs a forward and backward pass on its local mini-batch, producing local gradients $g_i$. To ensure every replica updates identically, the gradients must be globally averaged: $\bar{g} = \frac{1}{N}\sum_{i=1}^{N} g_i$. The operation that computes this sum and distributes the result to every GPU is called all-reduce.

The bandwidth-optimal algorithm for all-reduce is ring all-reduce. It arranges the $N$ GPUs in a logical ring and proceeds in two phases: reduce-scatter (each GPU accumulates partial sums) followed by all-gather (each GPU broadcasts its completed chunk). The total data transferred per GPU is:

$$V_{\text{allreduce}} = \frac{2(N-1)}{N} \cdot |\theta|$$

where $|\theta|$ is the total parameter count (all-reduce volume, measured in elements). For large $N$, this approaches $2|\theta|$ — the volume is essentially independent of the number of GPUs. This is what makes ring all-reduce scalable: adding more GPUs does not increase the per-GPU communication burden.

DDP: Overlapping Communication with Computation

Naive data parallelism waits for the backward pass to finish, then runs all-reduce, then runs the optimizer step. PyTorch's Distributed Data Parallel (DDP) eliminates this serialization by overlapping communication with computation. DDP groups parameters into buckets and triggers the all-reduce for each bucket as soon as its gradients are ready — while earlier layers are still computing their gradients. Since the backward pass proceeds from the last layer to the first, the final-layer gradients are ready first and can be communicated immediately.

Gradient Accumulation as Communication Reduction

Chapter 6 introduced gradient accumulation as a way to simulate larger batch sizes on a single GPU. In the distributed setting, it serves a second purpose: reducing communication. If each GPU accumulates gradients over $K$ micro-batches before triggering the all-reduce, the communication frequency drops by $K\times$ while the effective global batch size becomes $K \cdot N \cdot B_{\text{micro}}$. BLOOM's training used $K = 2$ gradient accumulation steps across 384 GPUs, yielding a global batch size of 2,048 sequences.

Batch Size Scaling

Larger global batch sizes enable faster training (more tokens per step) but require careful learning rate adjustment. The linear scaling rule says: if you multiply the batch size by $k$, multiply the learning rate by $k$. The square root rule, more conservative, scales the learning rate by $\sqrt{k}$. Both require a warm-up period — the first few hundred steps use a lower learning rate to avoid early instability. LLaMA 3 405B used a global batch size of 4M tokens across 16,384 GPUs with a gradual ramp from 8M to 16M tokens during the first phase of training.

The following shows a minimal DDP training loop with gradient accumulation.


import torch
import torch.distributed as dist
from torch.nn.parallel import DistributedDataParallel as DDP
from torch.utils.data.distributed import DistributedSampler

def train_ddp(rank, world_size, model, dataset, accum_steps=4):
    dist.init_process_group("nccl", rank=rank, world_size=world_size)
    model = model.to(rank)
    model = DDP(model, device_ids=[rank])  # wrap model for gradient sync
    sampler = DistributedSampler(dataset, num_replicas=world_size, rank=rank)
    loader = torch.utils.data.DataLoader(dataset, batch_size=8, sampler=sampler)
    optimizer = torch.optim.AdamW(model.parameters(), lr=3e-4)

    for epoch in range(num_epochs):
        sampler.set_epoch(epoch)  # shuffle differently each epoch
        for step, batch in enumerate(loader):
            input_ids = batch["input_ids"].to(rank)        # (B, seq_len)
            labels = batch["labels"].to(rank)               # (B, seq_len)

            with model.no_sync() if (step + 1) % accum_steps != 0 else nullcontext():
                loss = model(input_ids, labels=labels).loss  # forward + backward
                (loss / accum_steps).backward()              # scale gradient

            if (step + 1) % accum_steps == 0:
                # all-reduce happens here automatically via DDP
                torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
                optimizer.step()
                optimizer.zero_grad()

Model Parallelism

Data parallelism requires every GPU to hold a full copy of the model. A 70B parameter model in BF16 consumes 140GB just for the weights — already exceeding the 80GB of an A100. When the model does not fit on one device, we must split the model itself across GPUs. There are two orthogonal strategies: tensor parallelism (split within a layer) and pipeline parallelism (split across layers).

Tensor Parallelism: Megatron-LM

Tensor parallelism, introduced in Megatron-LM (Shoeybi et al., 2019), splits individual weight matrices across GPUs. The key insight is that the two linear layers in the MLP — and the QKV projections in attention — can be partitioned so that most computation happens independently, with a single all-reduce synchronizing the result.

Consider a linear layer $Y = XW$ (matrix multiply, $\mathbb{R}^{b \times d} \times \mathbb{R}^{d \times d} \to \mathbb{R}^{b \times d}$). We can split $W$ in two ways:

Column-parallel: Split $W$ along the output dimension into $t$ slices: $W = [W_1 | W_2 | \ldots | W_t]$, where each $W_i \in \mathbb{R}^{d \times d/t}$. Each GPU $i$ computes $Y_i = XW_i$ independently. The outputs $Y_i$ are different columns of the full result — no communication is needed if the next operation (like GeLU) applies element-wise.

Row-parallel: Split $W$ along the input dimension: $W = [W_1; W_2; \ldots; W_t]^\top$, where each $W_i \in \mathbb{R}^{d/t \times d}$. Each GPU computes $Y_i = X_i W_i$ on its local slice of the input, producing a partial sum. An all-reduce sums these partial results to get the final output $Y = \sum_i Y_i$ (all-reduce, all GPUs receive $Y \in \mathbb{R}^{b \times d}$).

Megatron-LM uses conjugate operators $f$ and $g$ to handle the forward and backward passes. In the forward pass, $f$ is the identity (input is replicated) and $g$ performs the all-reduce. In the backward pass, the roles reverse. This elegant design means the MLP requires exactly one all-reduce in the forward pass and one in the backward pass. Self-attention follows the same pattern: Q, K, V projections are column-parallel (each GPU handles a subset of attention heads), attention is computed independently, and the output projection is row-parallel with an all-reduce.

The following implements a simplified column-parallel linear layer.


import torch
import torch.nn as nn
import torch.distributed as dist

class ColumnParallelLinear(nn.Module):
    """Megatron-style column-parallel linear layer.
    Splits weight along output dimension across TP group."""

    def __init__(self, in_features, out_features, tp_group):
        super().__init__()
        self.tp_group = tp_group
        self.tp_size = dist.get_world_size(group=tp_group)
        self.local_out = out_features // self.tp_size
        # Each GPU holds a column slice: (in_features, out_features / tp_size)
        self.weight = nn.Parameter(torch.randn(in_features, self.local_out) * 0.02)
        self.bias = nn.Parameter(torch.zeros(self.local_out))

    def forward(self, x):
        # x: (batch, seq_len, in_features) — replicated on all TP ranks
        # Local matmul: no communication needed
        out = x @ self.weight + self.bias  # (batch, seq_len, local_out)
        return out  # each GPU holds different columns of the full output

class RowParallelLinear(nn.Module):
    """Row-parallel: splits weight along input dimension, all-reduces output."""

    def __init__(self, in_features, out_features, tp_group):
        super().__init__()
        self.tp_group = tp_group
        self.tp_size = dist.get_world_size(group=tp_group)
        self.local_in = in_features // self.tp_size
        self.weight = nn.Parameter(torch.randn(self.local_in, out_features) * 0.02)
        self.bias = nn.Parameter(torch.zeros(out_features))

    def forward(self, x):
        # x: (batch, seq_len, local_in) — each GPU holds its column slice
        out = x @ self.weight             # (batch, seq_len, out_features) partial
        dist.all_reduce(out, group=self.tp_group)  # sum partial results
        return out + self.bias            # (batch, seq_len, out_features) complete

Pipeline Parallelism

While tensor parallelism splits within a layer, pipeline parallelism splits across layers. The model's $L$ layers are divided into $p$ stages, each assigned to a different GPU. Data flows sequentially through the stages, like an assembly line. The challenge is the pipeline bubble: during startup and shutdown, some stages sit idle.

GPipe (Huang et al., NeurIPS 2019) splits each mini-batch into $m$ micro-batches and runs all forward passes first, then all backward passes. The bubble fraction is:

$$B_{\text{GPipe}} = \frac{p - 1}{m + p - 1}$$

where $p$ is the number of pipeline stages and $m$ is the number of micro-batches. With $p = 4$ stages and $m = 32$ micro-batches, the bubble is only $3/35 \approx 8.6\%$. However, GPipe stores activations for all $m$ micro-batches simultaneously, consuming significant memory.

The 1F1B schedule (one-forward-one-backward, Narayanan et al., SC 2021) interleaves forward and backward passes. After the warmup phase, each stage alternates between forwarding the next micro-batch and backwarding a completed one. This limits in-flight activations to $p$ micro-batches instead of $m$, dramatically reducing memory at the cost of a slightly different bubble formula: $B_{\text{1F1B}} = (p - 1) / m$.

ZeRO: Zero Redundancy Optimizer

Data parallelism duplicates the entire training state on every GPU: parameters, gradients, and optimizer states. For a model with $\Psi$ parameters using mixed-precision AdamW, the per-parameter memory cost is: 2 bytes for BF16 parameters + 2 bytes for BF16 gradients + 4 bytes for FP32 parameter copy + 4 bytes for FP32 first moment + 4 bytes for FP32 second moment = 16 bytes per parameter. For a 7B model, that is 112GB — replicated on every single GPU.

The ZeRO optimizer (Rajbhandari et al., SC 2020) eliminates this redundancy by partitioning instead of replicating. Across $N$ data-parallel GPUs:

StageWhat is partitionedMemory per GPU7B on 8 GPUs
No ZeRONothing (full replicas)$16\Psi$112 GB
Stage 1Optimizer states$(4 + 12/N)\Psi$38.5 GB
Stage 2+ Gradients$(2 + 14/N)\Psi$26.25 GB
Stage 3+ Parameters$16\Psi / N$14 GB

The communication cost is modest. Stages 1 and 2 communicate the same volume as standard DDP all-reduce: $2|\theta|$ elements. Stage 3 adds an extra all-gather of parameters before each forward pass, bringing the total to $3|\theta|$ — a 1.5× increase over DDP.

ZeRO-Infinity extends this further by offloading partitions to CPU memory or NVMe SSDs. With CPU offloading, a single DGX node with 8 GPUs can train models with over a trillion parameters — limited by CPU memory (typically 1-2TB) rather than GPU memory.

FSDP: PyTorch-Native ZeRO Stage 3

PyTorch's Fully Sharded Data Parallel (FSDP) implements ZeRO Stage 3 natively. The newer FSDP2 API (TorchTitan, 2024) uses per-parameter DTensor sharding, achieving 7% lower peak memory than FSDP1 and composing cleanly with tensor and pipeline parallelism.


import torch
from torch.distributed.fsdp import fully_shard, MixedPrecisionPolicy

def setup_fsdp_training(model, lr=3e-4):
    """Configure FSDP2 (ZeRO Stage 3) with mixed precision."""
    mp_policy = MixedPrecisionPolicy(
        param_dtype=torch.bfloat16,
        reduce_dtype=torch.float32,   # all-reduce in FP32 for stability
    )
    # Shard each transformer layer independently for optimal communication
    for layer in model.layers:
        fully_shard(layer, mp_policy=mp_policy)
    # Shard the full model (handles embeddings and output head)
    fully_shard(model, mp_policy=mp_policy)

    optimizer = torch.optim.AdamW(model.parameters(), lr=lr, weight_decay=0.1)
    return model, optimizer

Key Insight

ZeRO does not change the mathematical computation at all — every gradient, parameter update, and forward pass produces exactly the same result as naive data parallelism. It only changes where things are stored. The optimizer states, gradients, and parameters still exist in full — they are just partitioned across GPUs instead of redundantly replicated. Communication fills in the gaps just-in-time.

3D Parallelism

Data parallelism, tensor parallelism, and pipeline parallelism are not competing alternatives — they are orthogonal dimensions that compose. A cluster of GPUs is logically organized as a 3D grid where each axis handles a different bottleneck: TP handles within-layer memory, PP handles cross-layer memory, and DP handles throughput scaling.

The rule of thumb for choosing parallelism degrees: TP ≤ 8 to stay within a single NVLink-connected node (900 GB/s on H100), since tensor parallelism's all-reduces demand high bandwidth. PP spans nodes over InfiniBand (400 Gbps), since pipeline parallelism only sends activations between adjacent stages — lower volume, more latency-tolerant. DP fills the remaining dimension, scaling throughput with gradient all-reduces that are bandwidth-optimal via ring all-reduce.

DeepSeek V3 is a fascinating exception: it used *zero* tensor parallelism because the H800's NVLink bandwidth (160 GB/s, export-restricted) was only 3.2× faster than InfiniBand — not enough to justify the TP communication overhead. Instead, they used a novel DualPipe bidirectional pipeline schedule and expert parallelism for MoE routing, achieving 42.9% MFU while spending only $5.6M on the entire training run.

Training Infrastructure

GPU Cluster Topology

Modern GPU clusters have a two-tier bandwidth hierarchy. Within a node, 8 GPUs communicate over NVLink via NVSwitch — 900 GB/s bidirectional on H100 (DGX H100). Across nodes, communication travels over InfiniBand or RoCE — NDR InfiniBand provides 400 Gbps per port, with DGX H100 nodes having 10 ConnectX-7 NICs for approximately 1 TB/s peak bidirectional bandwidth per node. The 18× bandwidth gap between intra-node and inter-node communication is the fundamental constraint that shapes parallelism decisions.

Activation Checkpointing

During training, the forward pass stores intermediate activations for use in the backward pass. For a transformer with $L$ layers, this can consume enormous memory — a single layer with hidden size 4096 and sequence length 2048 stores roughly 830MB of activations in BF16. Across 80 layers, that is 66GB just for activations.

Activation checkpointing (Chen et al., 2016) trades compute for memory: discard most activations during the forward pass and recompute them during the backward pass. Full checkpointing saves only the inputs to each segment of $\sqrt{L}$ layers, achieving $O(\sqrt{L})$ memory at the cost of 30-40% additional compute. Selective checkpointing (Korthikanti et al., MLSys 2023) is smarter: it recomputes only cheap, bandwidth-bound operations (layer normalization, GeLU, dropout) while keeping the expensive matmul results. This achieves 50-60% memory savings at only 10-15% extra compute.

FlashAttention's Role

FlashAttention (Chapter 37) eliminates the $O(n^2)$ memory footprint of the attention matrix by computing attention in tiled blocks within GPU SRAM, never materializing the full $n \times n$ matrix in HBM. This is orthogonal to activation checkpointing and stacks with it — together, they make 128K+ context training feasible on current hardware.

Monitoring: MFU

Model FLOP Utilization (MFU) measures what fraction of a GPU cluster's theoretical peak compute is actually used for model computation:

$$\text{MFU} = \frac{6 \cdot N \cdot \text{tokens/sec}}{\text{num\_GPUs} \cdot P_{\text{peak}}}$$

where the factor 6 comes from the $\approx 6N$ FLOPs per token for a model with $N$ parameters (2N forward + 4N backward), and $P_{\text{peak}}$ is the peak FLOPS of one GPU (989 TFLOPS for H100 BF16). Achieved MFU numbers: PaLM 540B on TPUv4 = 46.2%, Megatron-LM on H100 = 47-48%, MegaScale (ByteDance, 12,288 GPUs) = 55.2%, LLaMA 3 405B = ~40%. Anything above 35% is considered good; above 50% is excellent.

Chinchilla-Optimal Data Planning

Given a compute budget, how should you split it between model size and training data? Chapter 10 derived the Chinchilla scaling laws (Hoffmann et al., 2022). The headline result is that the compute-optimal token count scales linearly with model size:

$$D^* \approx 20N$$

where $D^*$ is the optimal number of training tokens and $N$ is the parameter count (Chinchilla-optimal tokens, a dimensionless ratio). The total training compute is approximately $C \approx 6ND$ FLOPs (training compute estimate, where the factor 6 accounts for forward and backward passes). A 7B model is Chinchilla-optimal at 140B tokens; a 70B model at 1.4T tokens.

Beyond Chinchilla: The Inference-Optimal Regime

Chinchilla optimizes for training compute — but ignores inference cost. A model that will serve billions of requests over its lifetime should be smaller and trained longer, because each inference token is cheaper with a smaller model. This insight, formalized by Sardana and Frankle (ICML 2024), drove the dramatic overtraining in the LLaMA family:

Data Epochs and Repetition

When token budgets exceed available unique data, repetition becomes necessary. Empirically, a second pass over the data is noticeably worse than the first — loss curves flatten and downstream task gains diminish. Several strategies mitigate this: augmenting with synthetic data (e.g., code generation, paraphrasing), mixing in new domain-specific corpora, and carefully scheduling the repetition so that high-quality data is seen more often.

Curriculum Learning

Rather than presenting data randomly, curriculum learning orders examples from easy to hard — or, more commonly in LLM training, from general web data to higher-quality curated data. LLaMA 3 used a phased approach: the bulk of training used a broad web mixture, followed by an "annealing" phase on a smaller, curated dataset with higher code and math concentration. This late-stage data mix shift improved reasoning benchmarks significantly without degrading general capabilities.

Training Failures and Recovery

At the scale of thousands of GPUs running for months, hardware failures are not exceptional events — they are the default operating condition. LLaMA 3 405B experienced 466 job interruptions over 54 days of training on 16,384 H100 GPUs. That is roughly one failure every three hours. Of these, 419 were unplanned: 35% GPU failures, 17% HBM3 memory errors, and the remainder a mix of network, host, and software issues. Despite this, Meta achieved over 90% effective training time with only 3 manual interventions — the rest was handled by automated detection and recovery.

Loss Spikes

Loss spikes — sudden, sharp increases in the training loss — are a persistent challenge in large-scale training. PaLM 540B experienced approximately 20 loss spike events. The standard mitigation is to roll back to a checkpoint taken ~100 steps before the spike, skip the 200-500 data batches that triggered it, and resume. The root cause is often never fully determined, though candidates include data quality issues, numerical instability in specific micro-batches, and rare hardware errors.

The following implements a simple spike detector that monitors the training loss.


import torch
from collections import deque

class LossSpikeDetector:
    """Detect and respond to loss spikes during large-scale training."""

    def __init__(self, window_size=100, z_threshold=4.0):
        self.window = deque(maxlen=window_size)
        self.z_threshold = z_threshold
        self.spike_count = 0

    def update(self, loss_value: float, step: int) -> bool:
        """Returns True if a spike is detected."""
        if len(self.window) < 20:  # need enough history
            self.window.append(loss_value)
            return False

        mean = sum(self.window) / len(self.window)
        var = sum((x - mean) ** 2 for x in self.window) / len(self.window)
        std = var ** 0.5 + 1e-8

        z_score = (loss_value - mean) / std
        is_spike = z_score > self.z_threshold

        if is_spike:
            self.spike_count += 1
            print(f"[Step {step}] SPIKE DETECTED: loss={loss_value:.4f}, "
                  f"mean={mean:.4f}, z={z_score:.1f} (spike #{self.spike_count})")
        self.window.append(loss_value)
        return is_spike

Checkpointing at Scale

Checkpoint frequency is a balance: too rare and a failure wastes days of compute, too frequent and the I/O overhead slows training. BLOOM checkpointed every 3 hours (2.3TB per checkpoint across 384 GPUs). More recently, ByteDance's ByteRobust system demonstrated every-step checkpointing with less than 0.9% overhead by using asynchronous writes and memory-mapped files — essentially making checkpoint cost negligible.

Silent Data Corruption

Silent Killers

Silent data corruption (SDC) — a GPU computes the wrong result without raising an error — is the most insidious failure mode in large-scale training. LLaMA 3 experienced 6 SDC-attributed interruptions in 54 days on 16,384 H100 GPUs. Google reported SDC events every 1-2 weeks during Gemini training. A single bitflip in a gradient can propagate through the optimizer state and permanently corrupt the model. Detection requires comparing activation checksums across redundant data-parallel replicas — an expensive operation that most frameworks skip by default.

Straggler Mitigation

In pipeline parallelism, the slowest stage determines the throughput of the entire pipeline. A straggler — a GPU running slower due to thermal throttling, network congestion, or degraded memory — can reduce cluster throughput by 10-20%. Mitigations include monitoring per-GPU step times, proactive node replacement when performance drops below a threshold, and redundant computation where critical stages run on backup GPUs that can take over instantly.

The engineering required to sustain >90% effective training time across tens of thousands of GPUs is arguably as challenging as the model architecture itself.
