Chapter 12

Long-Context and Memory Augmentation

In 2023, a 4,096-token context window was standard. By early 2026, Google's Gemini 3 Pro accepts 10 million tokens — a 2,400× expansion in three years. Yet Chroma's "Context Rot" study tested 18 frontier LLMs and found that none of them process their context uniformly: performance degrades as input grows, information in the middle gets lost, and a single distractor sentence can derail retrieval from a million-token haystack. The context window expanded, but the model's ability to *use* it did not keep pace. This chapter traces the engineering and algorithmic innovations that extend context — RoPE interpolation, FlashAttention, Ring Attention — and the parallel development of external memory systems — RAG, knowledge graphs, neural memory modules — that give transformers access to information beyond any fixed window.

The Context Window Scaling Problem

Self-attention computes pairwise interactions between every token in the sequence. Double the sequence length and you quadruple the compute: attention costs $O(n^2 d)$ where $n$ is the sequence length and $d$ is the head dimension (matrix multiplication, $\mathbb{R}^{n \times d} \times \mathbb{R}^{d \times n} \to \mathbb{R}^{n \times n}$). Processing 128K tokens costs 16× more than processing 32K tokens — and 1,024× more than processing 4K tokens. That is just the FLOP cost. Memory tells a worse story.

The Quadratic Wall

KV Cache Memory

During autoregressive generation, the model stores key and value tensors for every previous token to avoid recomputation (Chapter 6). This KV cache grows linearly with sequence length: each new token adds $2 \times L \times H \times d_k$ parameters (2 for K and V, $L$ layers, $H$ heads, $d_k$ head dimension — a counting product, $\mathbb{R}^{2 \times L \times H \times d_k}$). For Llama-3 70B in FP16, the KV cache at 128K tokens consumes roughly 40 GB of VRAM — more than the entire model weights of many smaller LLMs. At 1M tokens it would exceed 300 GB, requiring multi-GPU inference just for the cache.

Positional Encoding Extrapolation

RoPE (Chapter 4) encodes position by rotating the query and key vectors — each dimension pair rotates at a different frequency determined by the base wavelength. When a model trained on 4K tokens sees position 8,000 for the first time, some of these rotation frequencies produce angles the model has never encountered. The attention scores become unpredictable, and generation quality collapses. This is not a gradual degradation — it is a cliff. A model that writes coherent prose at 4,095 tokens may produce gibberish at 4,097.

Long-Context Benchmarks

How do we know whether a model actually *uses* its long context, rather than just accepting it? The field has developed increasingly challenging benchmarks:

Needle in a Haystack (NIAH) hides a fact inside a long document and asks the model to retrieve it. Most frontier models pass NIAH at all positions and lengths — it is too easy. The model just needs to find a specific sentence.

RULER (NVIDIA, 2024) adds multiple needles, distractor needles, and aggregation tasks. Despite perfect NIAH scores, most models fail RULER as input length increases. The gap reveals that "accepts N tokens" does not mean "reasons over N tokens."

Sequential-NIAH (2025) requires extracting needles in the correct temporal or logical order. The best-performing model achieved only 63.15% accuracy — a humbling result given that frontier models claim 1M+ token windows.

Key Insight

A model's advertised context window is not its effective context window. Chroma's Context Rot study found that models paradoxically perform *better* on shuffled haystacks than logically coherent ones — suggesting that models rely on superficial pattern matching rather than deep reading. The practical lesson: where you place critical information in the prompt matters as much as whether it fits.

Extending Context Windows

The quadratic wall is real, but three classes of innovation have pushed through it: positional encoding tricks that let models generalize beyond their training length, attention algorithms that reduce compute and memory, and distributed techniques that spread long sequences across multiple GPUs.

Positional Interpolation (PI)

The simplest approach: if the model was trained on positions 0 to $L$, and we want it to handle positions 0 to $L' = sL$, we scale every position index down by $s$. Position 8,000 becomes position $8000/s$, which falls within the training range. Chen et al. (2023) showed this works surprisingly well with minimal fine-tuning.

The cost: local resolution suffers. Positions that were one step apart in the original encoding are now $1/s$ steps apart in the interpolated encoding. For $s = 4$, adjacent tokens share nearly identical positional embeddings, which can blur short-range attention patterns.

NTK-Aware Scaling

Not all RoPE frequencies deserve equal interpolation. The key insight from Neural Tangent Kernel theory: high-frequency dimensions encode local relationships (adjacent tokens), while low-frequency dimensions encode global position. NTK-aware scaling applies less interpolation to high frequencies (preserving local resolution) and more to low frequencies (extending range). This distributes the "interpolation pressure" more intelligently, but the optimal base frequency is difficult to find and usually requires expensive grid search.

YaRN: The Production Standard

YaRN (Yet another RoPE extensioN, Peng et al., 2023) combines the best ideas. It uses NTK-by-parts interpolation — a piecewise linear function that smoothly transitions between no interpolation (for the highest frequencies) and full interpolation (for the lowest). On top of this, YaRN adds a temperature scaling factor to the attention logits that compensates for the increased dot-product magnitudes at longer sequences.

YaRN requires 10× fewer tokens and 2.5× fewer training steps than position interpolation. Nearly every major open-weight model family — Qwen, DeepSeek, Llama — uses YaRN for context extension in production.

LongRoPE: Per-Dimension Scaling

LongRoPE (Microsoft, 2024) generalizes YaRN further by assigning an independent scaling factor to *every* positional dimension. Where position interpolation uses one scalar $s$, and YaRN uses a piecewise function over frequency bands, LongRoPE uses a learned vector $\mathbf{s} \in \mathbb{R}^{d/2}$ — one scale per rotation pair. This maximizes long-range stability while minimizing short-range degradation. LongRoPE extends context to 2M+ tokens.

Here is a PyTorch implementation showing the core idea of RoPE interpolation and YaRN-style scaling:


import torch
import math

def rope_frequencies(d_model, base=10000.0):
    """Standard RoPE frequencies: theta_i = base^(-2i/d)."""
    dim = d_model // 2
    freqs = 1.0 / (base ** (torch.arange(0, dim).float() / dim))
    return freqs                                              # (d/2,)

def apply_rope_with_scaling(q, k, positions, freqs, scale_factor=1.0,
                            method="pi"):
    """Apply RoPE with different interpolation methods.

    Args:
        q, k: (batch, seq_len, d_model) query and key tensors
        positions: (seq_len,) position indices
        freqs: (d/2,) base frequencies
        scale_factor: context extension ratio (e.g., 4.0 for 4x extension)
        method: "pi" (uniform), "ntk" (NTK-aware), or "yarn" (YaRN)
    """
    dim = freqs.shape[0]
    if method == "pi":
        # Position Interpolation: scale down all positions uniformly
        positions = positions / scale_factor
    elif method == "ntk":
        # NTK-Aware: adjust the base, not positions
        new_base = 10000.0 * scale_factor ** (dim / (dim - 2))
        freqs = 1.0 / (new_base ** (torch.arange(0, dim).float() / dim))
    elif method == "yarn":
        # YaRN: band-wise — low dims unscaled, high dims interpolated
        alpha, beta = 1.0, 32.0
        ramp = torch.clamp((torch.arange(dim).float() - alpha) / (beta - alpha),
                           0, 1)
        # ramp=0 → no interpolation; ramp=1 → full interpolation
        freqs = freqs / (1 + (scale_factor - 1) * ramp)

    # Build rotation angles: position × frequency
    angles = positions.unsqueeze(-1) * freqs.unsqueeze(0)    # (S, d/2)
    cos_a, sin_a = angles.cos(), angles.sin()                 # (S, d/2)

    # Apply rotation to query and key (split into pairs)
    q1, q2 = q[..., :dim], q[..., dim:]                      # (B, S, d/2)
    k1, k2 = k[..., :dim], k[..., dim:]
    q_rot = torch.cat([q1 * cos_a - q2 * sin_a,
                       q1 * sin_a + q2 * cos_a], dim=-1)     # (B, S, d)
    k_rot = torch.cat([k1 * cos_a - k2 * sin_a,
                       k1 * sin_a + k2 * cos_a], dim=-1)
    return q_rot, k_rot

Sliding Window + Global Attention

Longformer (Beltagy et al., 2020) replaces full $O(n^2)$ attention with local sliding-window attention ($O(n \cdot w)$ for window size $w$) plus global attention on a few designated tokens. Each token attends to its $w$ nearest neighbors — capturing local context efficiently — while global tokens (e.g., the [CLS] token or task-specific markers) attend to all positions.

Mistral 7B uses a sliding window of 4,096 tokens in its attention layers, allowing it to process sequences much longer than 4K tokens by having information propagate across windows through multiple layers. After $L$ layers with window $w$, each token's receptive field spans $L \times w$ positions — the window stacks across depth.

LongLoRA: Efficient Fine-Tuning for Long Context

LongLoRA (Chen et al., ICLR 2024) makes a surprising observation: while full attention is needed at inference, fine-tuning for context extension can use sparse attention. Its Shifted Sparse Attention (S²-Attn) splits the sequence into groups and applies attention within each group independently. In half of the attention heads, tokens are shifted by half the group size — ensuring information flows between adjacent groups. This is implementable in two lines of code during training. At inference, the model uses standard full attention.

LongLoRA extended Llama 2 7B from 4K to 100K context, and Llama 2 70B to 32K, on a single 8× A100 machine — a dramatic reduction in the hardware barrier for long-context fine-tuning.

Infrastructure: FlashAttention-3 and Ring Attention

Algorithmic innovations in attention extend how large a context the hardware can handle at each level of the memory hierarchy.

FlashAttention-3 (Dao et al., 2024) achieves 1.3 PFLOPs/s on H100 GPUs through three techniques: warp-specialization to overlap computation and data movement, interleaved block-wise matmul and softmax, and FP8 block quantization leveraging Hopper hardware support for low-precision arithmetic. FlashAttention is *exact* — it computes the same result as standard attention, but tiles the computation to minimize HBM reads/writes (Chapter 6).

Ring Attention distributes attention across multiple GPUs. The key and value tensors are split into chunks, each assigned to a different GPU. During execution, each GPU computes attention for its local query chunk while passing key-value chunks around a ring. After one full rotation, every GPU has attended to all positions.

Retrieval-Augmented Generation (RAG) as Memory

Instead of fitting everything into the context window, retrieve only what the model needs. Retrieval-Augmented Generation (RAG) treats an external knowledge base as an on-demand memory: the model queries it at generation time, pulling relevant documents into its context before answering. This decouples the model's knowledge horizon from its context window.

The RAG Pipeline

Each stage involves design decisions that affect downstream quality. Chunk size is the most impactful: smaller chunks (100-200 tokens) give higher retrieval precision but may lack sufficient context for the LLM to reason. Larger chunks (500-1000 tokens) provide more context per hit but dilute the relevant signal with surrounding text. Overlap between chunks (typically 10-20%) ensures that information split across a chunk boundary is still retrievable.

Late interaction models like ColBERT (Chapter 13) compute similarity between every query token and every passage token — more expensive than single-vector cosine similarity, but dramatically more precise for nuanced queries. The efficiency-quality curve between bi-encoders (fast, approximate) and cross-encoders (slow, precise) motivates the two-stage retrieve-then-rerank architecture.

RAG vs. Long Context

With 1M+ token windows available, why bother with RAG? The trade-offs are nuanced:

DimensionLong ContextRAG
CostExpensive at scale — pay for all tokensCheaper — only retrieved chunks enter the LLM
FreshnessStatic snapshot — must reload for updatesDynamic — index updates without re-prompting
FaithfulnessContext rot degrades quality over long inputsFocused context, but retrieval errors compound

ReasoningCan see full document relationshipsMay miss cross-document connections

The 2026 consensus is neither pure approach wins. The strongest deployments use a hybrid architecture: vector retrieval identifies relevant passages, then a long-context model reasons across the retrieved context. This combination outperforms either approach alone in 7 of 8 enterprise use case categories studied. RAG narrows the haystack; long context handles the reasoning.

GraphRAG

GraphRAG (Microsoft, 2024) builds an entity-relation knowledge graph from the corpus before retrieval. Rather than matching raw text chunks to a query, the system traverses graph relationships — following edges from entities mentioned in the query to related facts, even if those facts share no lexical overlap with the query.

GraphRAG supports two retrieval modes. Local search starts from specific entities and follows their immediate neighborhood. Global search uses community detection on the knowledge graph, retrieving summaries of entire topic clusters. This is particularly powerful for questions that require synthesizing information across many documents — "What are the main themes in this corpus?" — where traditional chunk-based RAG fails because no single chunk contains the answer.

Agentic RAG

By 2026, Agentic RAG is the baseline for production AI systems. Instead of a fixed pipeline (retrieve → generate), an autonomous agent manages the retrieval process: it decides *when* to retrieve, *what queries* to issue, *whether* the retrieved results are sufficient, and *when to retry* with a reformulated query. The agent uses reflection (evaluating its own retrieval quality), planning (decomposing complex questions into sub-queries), and tool use (calling different retrieval backends for different content types).

Here is a minimal RAG retrieval implementation using PyTorch embeddings and cosine similarity:


import torch
import torch.nn.functional as F

def build_rag_index(chunks, embed_model):
    """Embed document chunks and build an in-memory index.

    Args:
        chunks: list of text strings (document fragments)
        embed_model: encoder that maps text → (d_embed,) vector
    Returns:
        index: (N, d_embed) tensor of normalized embeddings
    """
    embeddings = []
    for chunk in chunks:
        with torch.no_grad():
            vec = embed_model.encode(chunk)                    # (d_embed,)
        embeddings.append(vec)
    index = torch.stack(embeddings)                            # (N, d_embed)
    index = F.normalize(index, dim=-1)                         # L2 normalize
    return index

def retrieve_top_k(query, index, embed_model, k=5):
    """Retrieve top-k chunks by cosine similarity.

    Cosine similarity: q^T d (dot product of normalized vectors, yields scalar).
    """
    with torch.no_grad():
        q_vec = embed_model.encode(query)                      # (d_embed,)
        q_vec = F.normalize(q_vec.unsqueeze(0), dim=-1)        # (1, d_embed)
    scores = (q_vec @ index.T).squeeze(0)                      # (N,)
    top_k = scores.topk(k)                                     # top-k scores + indices
    return top_k.indices.tolist(), top_k.values.tolist()

RAG vs. Long Context: When to Use Which

Use long context when you need complete reasoning over a bounded, static document (a legal contract, a codebase, a research paper). Use RAG when your knowledge base is large, dynamic, or when you need to control cost at scale. Use the hybrid — RAG-augmented long context — when both conditions apply: large corpus, complex multi-document reasoning.

External Memory Mechanisms

RAG retrieves raw text and stuffs it into the context window. But what if the model could maintain a persistent, structured memory — something more like a database than a document dump? Several architectures give transformers access to information that extends beyond any fixed context window, each with a different trade-off between flexibility, integration depth, and compute cost.

MemGPT and Virtual Context Management

MemGPT (Packer et al., 2023) draws a direct analogy to operating system memory management. A computer has limited physical RAM and a larger virtual address space backed by disk. MemGPT gives the LLM a limited "physical context" (its actual window) and an unlimited "virtual context" (an external store). The model uses function calls to *page* information in and out — loading relevant facts from the virtual context when needed and archiving old context when the physical window fills up.

MemGPT has since evolved into the Letta framework — an open-source agent framework for building persistent agents with long-term memory. The core insight remains: the LLM itself decides what to remember and what to forget, using its own judgment about what is relevant to the current conversation.

KNN-LM: Nearest-Neighbor Interpolation

KNN-LM (Khandelwal et al., 2020) augments the language model at inference time without any retraining. The idea: build a datastore of (context, next-token) pairs from a large corpus, then at each generation step, interpolate between the LM's own distribution and a k-nearest neighbor distribution over the datastore.

The interpolated distribution is:

$$p(y | x) = \lambda \, p_{\text{kNN}}(y | x) + (1 - \lambda) \, p_{\text{LM}}(y | x)$$

where $\lambda$ is a mixing coefficient (scalar multiplication — $\lambda$ scales the kNN probability, $1 - \lambda$ scales the LM probability). The kNN distribution $p_{\text{kNN}}$ is computed from the distances between the current hidden state and the stored contexts in the datastore: closer neighbors get higher weight via a softmax over negative distances.

Here is a simplified KNN-LM interpolation step in PyTorch:


import torch
import torch.nn.functional as F

def knn_lm_step(hidden, lm_logits, datastore_keys, datastore_vals,
                k=64, temperature=10.0, lam=0.25):
    """Interpolate LM distribution with kNN distribution.

    Args:
        hidden: (d_model,) current hidden state from the LM
        lm_logits: (vocab_size,) raw logits from the LM head
        datastore_keys: (N, d_model) stored hidden states
        datastore_vals: (N,) corresponding next-token IDs
        k: number of nearest neighbors
        temperature: softmax temperature for kNN distances
        lam: interpolation weight for kNN (0 = pure LM, 1 = pure kNN)
    """
    # Find k nearest neighbors by L2 distance
    dists = torch.cdist(hidden.unsqueeze(0),
                        datastore_keys).squeeze(0)             # (N,)
    topk = dists.topk(k, largest=False)                        # smallest distances
    nn_dists = topk.values                                     # (k,)
    nn_tokens = datastore_vals[topk.indices]                    # (k,)

    # kNN distribution: softmax over negative distances
    knn_weights = F.softmax(-nn_dists / temperature, dim=0)    # (k,)
    knn_probs = torch.zeros(lm_logits.shape[0])                # (vocab_size,)
    knn_probs.scatter_add_(0, nn_tokens, knn_weights)

    # LM distribution
    lm_probs = F.softmax(lm_logits, dim=0)                    # (vocab_size,)

    # Interpolate: scalar × vector + scalar × vector
    final_probs = lam * knn_probs + (1 - lam) * lm_probs
    return final_probs

KNN-LM is powerful because it requires zero additional training — you build the datastore from any corpus and plug it in. Domain adaptation becomes trivial: swap the datastore, not the model.

RETRO: Retrieval Integrated into Training

RETRO (Borgeaud et al., 2022) goes further than KNN-LM by integrating retrieval into the model architecture itself. Every few layers, RETRO cross-attends over retrieved chunks from a database. The retrieval is performed using the model's own intermediate representations, meaning the model learns *what* to retrieve as part of its training process.

The result: a 7.5B parameter RETRO model matches the performance of a 175B parameter model that relies on parametric knowledge alone — a 25× parameter reduction. The external database acts as an efficient storage for factual knowledge, freeing the model's parameters to focus on reasoning and language generation.

Memorizing Transformers

Memorizing Transformers (Wu et al., 2022) cache the key-value activations from previous forward passes in an external memory. At each generation step, the model attends over both its current context and the cached activations from prior contexts using approximate nearest-neighbor search. This is functionally similar to extending the KV cache across conversation turns or even across documents — without the quadratic cost of attending to all prior tokens with full attention.

Titans: Neural Long-Term Memory

Titans (Google Research, January 2025) introduces a fundamentally different approach: a learned neural memory module that updates its weights *while processing the input*. The memory is a small multi-layer perceptron (MLP) whose parameters are modified by gradient descent at test time — not during training of the main model, but during each forward pass as the model reads new tokens.

The architecture pairs two systems. Attention acts as short-term memory — precise but bounded by the context window. The neural memory module acts as long-term memory — it compresses historical context into its weights, providing a persistent representation that extends far beyond the attention window. The model learns which information to store in the neural memory and which to handle with attention.

Titans outperform standard Transformers, Mamba-2, and Gated DeltaNet on language modeling, commonsense reasoning, genomics, and time series tasks. Google's follow-up MIRAS framework (December 2025) builds on Titans with several variants (YAAD, MONETA, MEMORA), establishing neural memory as a first-class component of next-generation sequence models.

KV Cache Optimization

Every approach discussed so far either extends the context window or adds external memory. A complementary strategy: make the KV cache itself cheaper to store. Recent innovations have dramatically reduced the memory cost of long-context inference:

  • FP8 quantization: Storing KV cache in 8-bit floating point reduces memory by 75% in Llama-3.1-8B without meaningful quality loss. Frameworks like vLLM support this natively.
  • TurboQuant (ICLR 2026): Pushes further with 3-bit keys and 2-bit values, achieving 2.6× compression per layer — storing each token in approximately 198 bytes versus 512 bytes in BF16.
  • PagedAttention (vLLM): Borrows paging from OS memory management — allocates KV cache in fixed-size blocks rather than contiguous arrays, reducing waste to under 4% and enabling 2-4× throughput improvement.
  • PagedEviction: Fine-grained structured pruning of KV cache blocks, dropping the least important blocks as context grows.

These optimizations compose with all the context extension techniques above. A model with YaRN context scaling, FlashAttention-3 for compute efficiency, FP8 KV cache quantization, and PagedAttention for memory management can serve 128K-token contexts at costs that were impractical for 8K contexts two years ago.
