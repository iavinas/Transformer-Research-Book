Chapter 13

Retrieval and Ranking with Transformers

Search "jaguar speed" on a lexical search engine and you will get results about both the animal and the car — the system matches words, not meaning. Replace the keyword matcher with a BERT encoder that maps the query and every document into the same vector space, and now "fast feline" retrieves the same pages as "jaguar speed" because both land near the same point in embedding space. This chapter walks through every layer of the modern retrieval stack: from classical BM25 to dense passage retrieval, from single-vector cosine similarity to ColBERT's token-level late interaction, from pointwise reranking to agentic RAG pipelines that decide what to retrieve, when, and how many times. By the end, you will understand why the fastest production systems in 2026 run hybrid sparse-dense retrieval with learned reranking — and how to build one.

The Information Retrieval Landscape

Before transformers touched search, two paradigms dominated: lexical matching and vector similarity. Understanding both is essential because production systems in 2026 still use both — hybrid retrieval outperforms either alone.

Classical Retrieval: TF-IDF and BM25

TF-IDF (Term Frequency–Inverse Document Frequency) represents each document as a sparse vector in vocabulary space. Each dimension corresponds to a word, and the value is the product of how often the word appears in the document (TF) and how rare it is across the corpus (IDF). Cosine similarity between query and document TF-IDF vectors gives a relevance score.

BM25 improves on TF-IDF with two corrections: it saturates the term frequency contribution (a word appearing 100 times is not 100× more relevant than one appearance), and it normalizes by document length (longer documents are not penalized just for containing more words). BM25 remains the default first-stage retriever in production — it is fast, requires no training, and handles exact-match queries (product IDs, code identifiers, rare proper nouns) that dense models miss.

The Semantic Gap

Lexical matching fails when the query and the relevant document use different words for the same concept. "Car" does not match "automobile." "How to fix a memory leak" does not match a document titled "Debugging heap exhaustion in Java." This semantic gap — the distance between surface-level word overlap and underlying meaning — is what neural retrieval models exist to close.

Dense Retrieval: The Core Idea

Map queries and documents into a shared vector space where semantic similarity corresponds to geometric proximity. Train the encoder so that a query about "fast felines" lands near a document about "jaguar speed records" even though they share no words. At retrieval time, compute the query vector, run approximate nearest neighbor search over the document vectors, and return the closest matches.

Bi-Encoder vs. Cross-Encoder

Two fundamental architectures compete for retrieval and ranking. The choice between them defines the efficiency-quality curve of every modern search system.

A bi-encoder encodes the query and the document *independently* with separate forward passes. The similarity score is a simple function of the two vectors — typically cosine similarity: $\text{sim}(q, d) = \frac{q^\top d}{\|q\|\|d\|}$ (dot product of normalized vectors, yielding a scalar). Because document vectors are pre-computed and cached, retrieval requires only one forward pass (for the query) plus a vector search. This is fast — milliseconds over millions of documents.

A cross-encoder concatenates the query and the document into a single input: $[\text{CLS}] \; q_1 \ldots q_m \; [\text{SEP}] \; d_1 \ldots d_n \; [\text{SEP}]$. The transformer attends jointly over both, capturing fine-grained token-level interactions. This is dramatically more accurate — the model sees how every query word relates to every document word — but requires a forward pass per (query, document) pair. Running a cross-encoder over a million documents is impractical.

The production solution: use both. A bi-encoder retrieves the top 100-1000 candidates from millions of documents in milliseconds. A cross-encoder reranks those candidates, spending the compute budget where it matters most. This cascade architecture — retrieve then rerank — is the backbone of every modern search system.

Dense Passage Retrieval (DPR)

DPR (Karpukhin et al., 2020) established the template for neural retrieval: two BERT encoders, contrastive training, and FAISS for nearest-neighbor search at scale.

Architecture

DPR uses two independent BERT encoders — one for questions, one for passages. Each produces a single vector from the [CLS] token's hidden state. The similarity between a question $q$ and passage $p$ is computed as their dot product: $\text{sim}(q, p) = E_Q(q)^\top E_P(p)$ (dot product yielding a scalar, where $E_Q(q) \in \mathbb{R}^d$ and $E_P(p) \in \mathbb{R}^d$).

Contrastive Training with Hard Negatives

The training objective pushes the positive passage closer to its question and the negative passages further away. For a question $q_i$ with positive passage $p_i^+$ and $N$ negative passages $\{p_{i,j}^-\}$, the loss is:

$$\mathcal{L} = -\log \frac{e^{\text{sim}(q_i, p_i^+)}}{e^{\text{sim}(q_i, p_i^+)} + \sum_{j=1}^{N} e^{\text{sim}(q_i, p_{i,j}^-)}}$$

This is the InfoNCE contrastive loss — structurally identical to the softmax cross-entropy, treating the positive as the correct class. The key to DPR's success is the choice of negatives: in-batch negatives (other questions' positives in the same mini-batch, cheap), BM25 negatives (passages that match lexically but are semantically wrong, hard), and gold negatives from the dataset. Hard negatives force the model to learn fine-grained distinctions rather than trivial ones.

Here is a DPR-style bi-encoder with contrastive training in PyTorch:


import torch
import torch.nn as nn
import torch.nn.functional as F

class BiEncoder(nn.Module):
    """DPR-style bi-encoder: independent query and passage encoders."""
    def __init__(self, d_model=768, d_embed=128):
        super().__init__()
        # In practice these would be BERT encoders; simplified here
        self.query_proj = nn.Linear(d_model, d_embed)
        self.passage_proj = nn.Linear(d_model, d_embed)

    def encode_query(self, q_hidden):
        return F.normalize(self.query_proj(q_hidden), dim=-1)   # (B, d_embed)

    def encode_passage(self, p_hidden):
        return F.normalize(self.passage_proj(p_hidden), dim=-1) # (B, d_embed)

def contrastive_loss(q_embeds, p_embeds, temperature=0.05):
    """InfoNCE contrastive loss with in-batch negatives.

    q_embeds: (B, d) normalized query embeddings
    p_embeds: (B, d) normalized passage embeddings
    Each q_embeds[i] is a positive match for p_embeds[i].
    All other passages in the batch serve as negatives.
    """
    # Similarity matrix: q_i^T p_j for all (i, j) pairs
    scores = q_embeds @ p_embeds.T / temperature               # (B, B)
    # Labels: q_i should match p_i (diagonal)
    labels = torch.arange(scores.size(0), device=scores.device) # (B,)
    loss = F.cross_entropy(scores, labels)
    return loss

FAISS: Billion-Scale Nearest Neighbor Search

Once passage vectors are computed, retrieval reduces to a nearest-neighbor search: find the $k$ passages whose vectors are closest to the query vector. Exact search over billions of vectors is impractical — a brute-force scan of 1 billion 768-dimensional vectors takes seconds per query. FAISS (Facebook AI Similarity Search) solves this with approximate nearest neighbor algorithms.

The two most common FAISS index types are IVF (Inverted File Index), which partitions the vector space into clusters and only searches the nearest clusters to the query, and PQ (Product Quantization), which compresses each vector into a short code for memory efficiency. Combined as IVF-PQ, FAISS searches billions of vectors in under 10 milliseconds with recall above 90%.

DPR vs. BM25: When Each Wins

ColBERT: Late Interaction

Bi-encoders compress an entire document into one vector. This is efficient but lossy — a 500-word passage reduced to 128 numbers cannot preserve every nuance. Cross-encoders preserve every nuance but are too slow for retrieval. ColBERT (Khattab & Zaharia, SIGIR 2020) occupies the sweet spot: it retains per-token representations while still allowing pre-computation of document embeddings.

MaxSim: Token-Level Matching

ColBERT represents each query as a set of token embeddings $\{q_1, \ldots, q_m\}$ and each document as $\{d_1, \ldots, d_n\}$ (each $q_i, d_j \in \mathbb{R}^{d}$). The relevance score is the sum of maximum similarities:

$$\text{score}(q, d) = \sum_{i=1}^{m} \max_{j=1}^{n} \; q_i^\top d_j$$

For each query token $q_i$, find the document token $d_j$ it matches best (dot product, yielding a scalar), and sum these maximum similarities. This MaxSim operation captures token-level semantic matching — "speed" in the query aligns with "mph" in the document, while "feline" aligns with "jaguar" — without requiring joint attention between query and document.

ColBERTv2 and PLAID

ColBERTv2 (Santhanam et al., NAACL 2022) addresses the original ColBERT's storage cost — storing per-token embeddings for every document requires 6-10× more space than single-vector approaches. ColBERTv2 applies residual compression: each token embedding is quantized relative to its nearest centroid, storing only the small residual. Combined with denoised supervision (distillation from a cross-encoder teacher with hard negative mining), ColBERTv2 reduces storage by 6-10× while improving retrieval quality.

PLAID (Performance-optimized Late Interaction Driver) further accelerates ColBERT search. The key trick: treat each passage as a lightweight "bag of centroids" and use centroid interaction to quickly eliminate low-scoring passages before computing the full MaxSim. PLAID achieves 7× speedup on GPU and 45× on CPU over vanilla ColBERTv2, with retrieval latency of tens of milliseconds over 140 million passages.

ColPali: Vision-Native Document Retrieval

ColPali (ICLR 2025) extends late interaction to visual documents. Instead of extracting text from PDFs through OCR and layout analysis, ColPali directly embeds a *screenshot* of the document page using a Vision Language Model (PaliGemma). The "Col" in ColPali — as in ColBERT — means the model produces multiple vectors per page (one per visual patch), and scoring uses MaxSim across query tokens and visual patch embeddings.

ColPali dramatically outperforms text-extraction pipelines on visually rich documents: infographics, tables, charts, and mixed-language layouts. It eliminates the entire preprocessing stack — no OCR, no layout parsing, no text chunking — replacing it with a single end-to-end model.

Here is a ColBERT-style MaxSim scoring implementation in PyTorch:


import torch
import torch.nn.functional as F

def colbert_maxsim(q_embeddings, d_embeddings):
    """ColBERT MaxSim scoring between query and document token embeddings.

    Args:
        q_embeddings: (m, d) — m query token embeddings
        d_embeddings: (n, d) — n document token embeddings
    Returns:
        score: scalar — sum of max similarities per query token
    """
    # Token-level similarity matrix: q_i^T d_j for all pairs
    sim_matrix = q_embeddings @ d_embeddings.T                  # (m, n)
    # MaxSim: for each query token, find best-matching doc token
    max_sims = sim_matrix.max(dim=1).values                     # (m,)
    # Score = sum of max similarities
    return max_sims.sum()                                        # scalar

def colbert_batch_score(q_batch, d_batch):
    """Score a batch of query-document pairs with ColBERT MaxSim.

    Args:
        q_batch: (B, m, d) — B queries, m tokens each
        d_batch: (B, n, d) — B documents, n tokens each
    """
    # Batched similarity: each query against its corresponding document
    sim = torch.bmm(q_batch, d_batch.transpose(1, 2))          # (B, m, n)
    max_sims = sim.max(dim=2).values                            # (B, m)
    return max_sims.sum(dim=1)                                   # (B,)

Sparse Learned Retrieval

BM25 uses sparse vectors (vocabulary-sized, mostly zeros). Dense retrieval uses dense vectors (128-768 dimensions, no zeros). Sparse learned retrieval occupies the middle ground: it produces sparse, vocabulary-sized vectors — just like BM25 — but learns the weights using a neural network instead of counting term frequencies.

SPLADE: Sparse Lexical Expansion

SPLADE (SParse Lexical AnD Expansion model) uses BERT's masked language model head to produce a sparse document representation. For each token in the document, BERT predicts a distribution over the entire vocabulary. SPLADE takes the maximum prediction score across all positions for each vocabulary term, then applies a log-saturation function to produce the final sparse vector.

The critical capability: lexical expansion. A document about "Python memory management" will receive non-zero weights for related terms like "garbage collection," "heap," and "reference counting" — even if those exact words do not appear in the text. This bridges the semantic gap while maintaining compatibility with inverted indexes, meaning SPLADE can run on the same infrastructure as BM25.

uniCOIL: Learned Term Weights

uniCOIL (Unified COntextualized Inverted List) takes a simpler approach: it uses BERT to predict a single scalar weight for each token in the document, producing a sparse vector with non-zero entries only for tokens that actually appear. This makes it directly compatible with standard BM25 inverted indexes — just replace the BM25 term weights with learned ones. No new infrastructure needed.

Hybrid Retrieval and Reciprocal Rank Fusion

Reciprocal Rank Fusion (RRF) is the standard fusion algorithm. For a document $d$ appearing at rank $r_i$ in result list $i$:

$$\text{RRF}(d) = \sum_{i} \frac{1}{k + r_i(d)}$$

where $k = 60$ is a constant that prevents top-ranked documents from dominating. RRF is score-agnostic — it uses only rank positions, not raw scores — so it naturally handles the incomparable scales of BM25 scores, cosine similarities, and MaxSim sums without normalization.

Cross-Encoders for Reranking

After retrieval returns the top 100-1000 candidates, a cross-encoder reranks them. The accuracy improvement is dramatic: a MiniLM cross-encoder reranking BM25 results often outperforms a dense retriever alone, despite the BM25 first stage being purely lexical.

Architecture and Training Objectives

The cross-encoder takes the concatenated query-document pair $[\text{CLS}] \; q \; [\text{SEP}] \; d \; [\text{SEP}]$ and produces a relevance score from the [CLS] token through a linear head. Three training objectives are common:

Pointwise: Each (query, document) pair receives an independent relevance label (0/1 or graded). The model trains with binary cross-entropy or MSE. Simple but ignores relative ordering between documents.

Pairwise: Given a preferred and rejected document for the same query, the model learns to score the preferred one higher. Structurally similar to the Bradley-Terry reward model from Chapter 11.

Listwise: The model scores an entire list of documents simultaneously and optimizes a ranking metric (NDCG) directly. Set-Encoder (2024) extends this with permutation-invariant inter-passage attention — each document's score is informed by the other documents in the list.

MonoBERT and MiniLM Rerankers

MonoBERT uses a full BERT-base or BERT-large model for pointwise reranking. Accurate but slow — ~110ms per query at 100 candidates on a single GPU. MiniLM rerankers distill MonoBERT into a 6-layer model, achieving 70+ nDCG@10 on the TREC Deep Learning track at 3-5× lower latency. For most applications, MiniLM provides the best quality-latency trade-off.

Cross-encoders remain competitive against LLM-based rerankers even in 2026. A fine-tuned MiniLM reranker matches or exceeds GPT-4-class listwise reranking on standard benchmarks — at 100× lower cost and latency.

The Cascade Architecture

Here is a cross-encoder reranker implementation in PyTorch:


import torch
import torch.nn as nn

class CrossEncoderReranker(nn.Module):
    """Cross-encoder: joint query-document scoring via concatenated input."""
    def __init__(self, backbone, d_model=768):
        super().__init__()
        self.backbone = backbone  # BERT or similar encoder
        self.score_head = nn.Linear(d_model, 1)

    def forward(self, input_ids, attention_mask):
        """Score a batch of [CLS] q [SEP] d [SEP] inputs.

        input_ids: (B, S) — tokenized query-document concatenations
        attention_mask: (B, S) — padding mask
        Returns: (B,) relevance scores
        """
        h = self.backbone(input_ids, attention_mask)            # (B, S, d_model)
        cls_hidden = h[:, 0, :]                                  # (B, d_model)
        scores = self.score_head(cls_hidden).squeeze(-1)         # (B,)
        return scores

def rerank(reranker, query_ids, doc_ids_list, tokenizer, top_k=10):
    """Rerank candidate documents for a single query.

    query_ids: (Q,) tokenized query
    doc_ids_list: list of (D_i,) tokenized documents
    Returns: indices of top-k documents after reranking
    """
    scores = []
    for doc_ids in doc_ids_list:
        # Concatenate: [CLS] query [SEP] doc [SEP]
        input_ids = torch.cat([
            torch.tensor([tokenizer.cls_id]),
            query_ids,
            torch.tensor([tokenizer.sep_id]),
            doc_ids,
            torch.tensor([tokenizer.sep_id])
        ]).unsqueeze(0)                                          # (1, S)
        mask = torch.ones_like(input_ids)                        # (1, S)
        with torch.no_grad():
            score = reranker(input_ids, mask)                    # (1,)
        scores.append(score.item())
    ranked = sorted(range(len(scores)), key=lambda i: scores[i], reverse=True)
    return ranked[:top_k]

RAG Pipeline Architectures

Chapter 12 introduced RAG as memory augmentation. Here we examine how the retrieval pipeline itself has evolved from a simple "retrieve then generate" pattern into sophisticated agentic systems.

Naive RAG

The simplest RAG pattern: embed the user query, retrieve the top-$k$ chunks from a vector database, concatenate them into the LLM's context, and generate. This works for simple factual questions but fails on multi-hop reasoning, ambiguous queries, and cases where the initial retrieval misses the relevant documents.

Advanced RAG: Query Transformation

HyDE (Hypothetical Document Embeddings, Gao et al., 2023) flips the retrieval direction. Instead of embedding the query and searching for similar documents, HyDE first asks the LLM to generate a *hypothetical answer* — a plausible document that would contain the answer. This hypothetical document is then embedded, and similar *real* documents are retrieved. The insight: a hypothetical answer is already in "document space," making it a better search probe than a short query.

Multi-HyDE (2025) extends this by generating multiple diverse hypothetical answers, each capturing a different angle on the query. The retrieved results from all hypothetical documents are merged, improving recall on complex questions.

Other query transformations include query rewriting (the LLM rephrases the query for better retrieval), query decomposition (splitting a complex question into sub-questions that are retrieved independently), and step-back prompting (asking a more general question first to establish context).

Modular RAG

Modular RAG decomposes the pipeline into pluggable components: router (decides which retriever to use), retriever (sparse, dense, or both), reranker (ColBERT, cross-encoder, or LLM), and generator (with or without citation). Different queries route through different pipeline configurations — a factual question might use BM25 + cross-encoder, while a reasoning question might use dense retrieval + multi-hop chain.

Agentic RAG

By 2026, Agentic RAG is the production standard for complex information needs. The LLM agent controls the entire retrieval process: it decides *whether* to retrieve (maybe the answer is already in context), *what queries* to issue (potentially multiple reformulations), *which tools* to call (web search, vector DB, knowledge graph, SQL database), and *when to stop* (evaluating whether the retrieved context is sufficient to answer confidently).

Key Insight

The progression from naive to agentic RAG mirrors the progression from instruction-following (Chapter 11) to autonomous reasoning. In both cases, the system evolves from executing a fixed recipe to making dynamic decisions based on intermediate results. The retrieval pipeline becomes a tool that the agent wields, not a script it follows.

Multimodal and Cross-Modal Retrieval

Text retrieval is the foundation, but production systems increasingly need to search across modalities: find an image given a text description, retrieve code given a natural language query, or locate a specific moment in a video.

Image-Text Retrieval with CLIP

CLIP (Contrastive Language-Image Pre-training, Radford et al., 2021) trains a bi-encoder with one arm for images and one for text on 400 million image-text pairs scraped from the web. The contrastive objective pushes matching image-text pairs together in the shared embedding space while pushing non-matching pairs apart — the same InfoNCE loss used for DPR, but across modalities.

CLIP enables zero-shot image classification (embed a text label, find the closest image embeddings), text-to-image search (embed a description, search an image database), and image-to-text search (embed an image, find matching captions). The shared embedding space means any combination of modalities works without task-specific fine-tuning.

Code Retrieval

CodeBERT (Feng et al., 2020) pre-trains a transformer on paired code-text data (function implementations paired with docstrings). UniXcoder (Guo et al., 2022) extends this with a unified encoder-decoder that handles code generation, code summarization, and code retrieval in a single model. Both produce function-level embeddings that enable natural-language-to-code search — "find me a function that sorts a list in descending order" retrieves the right implementation.

Video Retrieval

Video retrieval adds the dimension of time. The challenge: a 10-minute video may contain the relevant information in a 5-second segment. Temporal aggregation strategies include mean-pooling frame embeddings (simple but loses temporal structure), learning temporal position encodings across frames, and hierarchical approaches that first embed segments and then compose them. CLIP-based models applied frame-by-frame with temporal attention layers form the current backbone of video retrieval systems.
